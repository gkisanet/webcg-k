# 🎨 AI SVG 생성 가이드 — 저비용 모델로 프로급 벡터 만들기

> **핵심 철학**: 최신 고성능 모델은 복잡한 공간 지각력과 수식 계산 능력이 뛰어나 정교한 결과물을 냅니다.
> 하지만 **저렴하고 가벼운 모델이라도** SVG의 작동 원리와 AI의 한계를 이해하고 프롬프트를 **'건축 설계도'**처럼
> 구조화하여 입력하면 고성능 모델 부럽지 않은 훌륭한 벡터 이미지를 뽑아낼 수 있습니다.

## 1. AI가 SVG로 잘 그리는 것 vs 못 그리는 것

### 🟢 잘 그리는 것 (가벼운 모델의 강점)

| 분류 | 예시 | 왜 잘 그리는가? |
|------|------|----------------|
| **기하학적 도형 기반 일러스트** | 스마트폰, 모니터, 톱니바퀴, 집 | `<rect>`, `<circle>`, `<polygon>` 등 기본 태그로 조립 가능 |
| **플랫(Flat) 디자인 아이콘** | UI/UX 에셋, 미니멀리즘 2D 아이콘 | 그림자/명암 없는 깔끔한 형태 |
| **대칭적인 로고 및 패턴** | 중심 좌표 기반 대칭 디자인 | 수학적 대칭 → AI가 좌표 계산 가능 |
| **데이터 차트 및 다이어그램** | 막대/파이 차트, 구조적 그래픽 | 수치 기반 → AI의 수학 능력 활용 |

### 🔴 못 그리는 것 (형태가 무너지는 한계점)

| 분류 | 증상 | 근본 원인 |
|------|------|----------|
| **복잡한 유기체** (사람 얼굴, 사실적 동물) | 눈이 3개, 형태 기괴하게 찌그러짐 | 베지에 곡선(`<path d="M... C...">`)을 수십~수백 개 조합할 때 **시각적 피드백이 없어** 공간적 맥락을 잃음 |
| **복잡한 원근법과 겹침(Z-index)** | 배경이 앞으로 튀어나옴, 눈이 머리 뒤로 숨음 | 코드로만 앞뒤 겹침 구현 → **논리적 순서 오류** 빈번 |
| **화려한 빛과 3D 그림자** | 코드 기하급수적 증가 + 오류 | 복잡한 메쉬 그라데이션/SVG 필터에서 붕괴 |
| **정교한 타이포그래피** | 글씨가 외계어로 변함 | `<path>`로 글자를 따서 그리면 실패. **반드시 `<text>` 태그 사용** |
| **토큰 한계로 인한 끊김** | `</svg>` 닫지 않고 출력 중단 | 복잡한 풍경 → 코드가 길어져 토큰 소진 |

> [!IMPORTANT]
> **AI는 눈으로 보지 않는다.** SVG를 생성하는 AI는 렌더링 결과를 볼 수 없습니다.
> 오직 **코드의 수학적 관계**만으로 형태를 추론합니다.
> 따라서 "예쁘게 그려줘"가 아닌 **"수학적으로 정확하게 조립하라"**고 지시해야 합니다.

## 2. 한계를 극복하는 5대 프롬프팅 가이드라인

> **비유**: AI에게 SVG를 그려달라고 하는 것은 **눈을 감고 레고 블록을 조립하라**는 것과 같습니다.
> 블록의 종류와 위치를 정확히 알려주면 훌륭한 결과물이 나옵니다.

### 💡 가이드 1. 복잡한 Path 대신 '기본 도형(Primitives)' 사용을 강제하라

> **🏆 가장 중요한 규칙!**

자유 곡선(`<path>`)을 모델의 상상력에 맡기면 형태가 망가집니다. 기본 도형의 조합을 강제해야 합니다.

| ❌ 나쁜 예 | ⭕ 좋은 예 |
|-----------|----------|
| "귀여운 고양이를 SVG로 그려줘." | "고양이 얼굴 아이콘을 만들어줘. 복잡한 `<path>` 태그 사용은 **엄격히 금지**하고, 반드시 `<circle>`, `<rect>`, `<ellipse>`, `<polygon>`, `<line>` 같은 **기본 기하학적 도형만** 조합해서 형태를 조립해." |

### 💡 가이드 2. 캔버스 크기(viewBox)를 단순한 숫자로 고정하라

AI가 비례와 좌표를 쉽게 계산할 수 있도록 **도화지의 크기와 중심점**을 명확히 박아줍니다.

```
⭕ "SVG의 viewBox를 0 0 400 400으로 고정하고,
    주요 객체는 캔버스의 정중앙(cx="200", cy="200")을 기준으로 대칭에 맞게 그려."
```

> [!TIP]
> **400×400, 500×500** 같은 깔끔한 숫자를 사용하면 AI가 좌표를 계산하기 훨씬 쉬워집니다.
> 1920×1080 같은 비대칭 해상도를 지정하면 중심 계산에서 실수할 확률이 높아집니다.

### 💡 가이드 3. 그룹화(`<g>`)와 주석을 요구하라 (Chain of Thought 효과)

코드를 짜기 전에 **주석으로 부위를 명시**하게 하면, AI 스스로 자신이 어디를 그리고 있는지 인지하게 되어 좌표 왜곡이 크게 줄어듭니다.

```xml
⭕ "배경, 머리, 눈, 코 등 각 구성 요소를 반드시 <g> 태그로 그룹화하고,
    그룹 코드 바로 위에 <!-- 🐱 고양이 왼쪽 눈 --> 처럼 직관적인 한글 주석을 달아줘."
```

> [!NOTE]
> 이것은 **Chain of Thought (사고의 사슬)** 효과입니다.
> 주석이 곧 AI의 "내면의 독백" 역할을 하여, 다음 코드 블록의 좌표 계산 정확도를 높입니다.

### 💡 가이드 4. 레이어(Z-index) 렌더링 순서를 텍스트로 명확히 지정하라

SVG는 코드가 아래에 적힐수록 화면의 **맨 앞(최상단)**에 덮어씌워지며 그려집니다.
가벼운 모델은 이 순서를 잘 헷갈리므로 명시해 주어야 합니다.

```
⭕ "코드는 렌더링 순서에 맞춰서
    1. 배경 → 2. 얼굴 윤곽 → 3. 눈/코/입 순서로 작성해
    (즉, 눈/코/입 코드가 맨 아래에 오도록)."
```

### 💡 가이드 5. 디자이너처럼 색상과 스타일을 명확히 제한하라

가벼운 모델의 미적 감각에 의존하지 마세요. **팔레트를 직접 지정**합니다.

```
⭕ "디자인 스타일은 '플랫(Flat) 디자인'으로 해줘.
    그라데이션이나 그림자는 절대 쓰지 말고,
    메인 컬러(#FFB347), 눈/코(#333333) 등 3~4가지 솔리드 컬러(Solid Color)만 사용해."
```

## 3. 실전 프롬프트 템플릿

> **사용법**: `[ ]` 부분만 원하는 내용으로 바꾸어 가벼운 모델(Gemini Flash, Claude Haiku, GPT-4o-mini 등)에 입력하세요.

```plaintext
너는 15년 경력의 UI/UX 아이콘 디자이너이자 수석 프론트엔드 개발자야.
아래의 [요구사항]과 [엄격한 SVG 작성 규칙]에 따라 완벽하게 렌더링되는 깨끗한 SVG 코드를 생성해 줘.

[요구사항]
- 주제: [커피 컵을 들고 있는 귀여운 로봇 얼굴 아이콘]
- 스타일: 모던하고 미니멀한 2D 플랫(Flat) 일러스트 디자인.
- 색상: [파스텔 톤 블루, 화이트, 다크 그레이 등 4가지 이하의 색상만 사용]

[엄격한 SVG 작성 규칙]
1. 캔버스 및 좌표: `<svg viewBox="0 0 500 500" xmlns="http://www.w3.org/2000/svg">`
   로 시작하며, 캔버스 정중앙을 기준으로 조화롭게 배치할 것.
2. 도형 제한 (가장 중요): 형태가 일그러지는 것을 막기 위해 복잡한 베지에 곡선
   (`<path>` 태그) 사용을 엄격히 최소화할 것. 반드시 `<rect>`, `<circle>`,
   `<ellipse>`, `<polygon>`, `<line>` 등 기본 기하학적 도형을 위주로 겹치고
   조립해서 대상을 표현할 것. (`<rect>`의 rx, ry 속성으로 모서리를 둥글게 하는
   것은 적극 권장함)
3. 그룹화 및 주석: 각 구성 요소(예: 로봇 머리, 안테나, 눈, 커피 컵 등)를
   `<g>` 태그로 묶어주고, 반드시 코드 위에 `<!-- 🤖 로봇 머리 -->`과 같이
   설계 의도를 명시하는 한글 주석을 달 것.
4. 레이어 순서: 배경 요소가 코드 상단에, 앞으로 튀어나오는 디테일한 요소
   (눈, 코 등)가 코드 하단에 오도록 Z-index 렌더링 순서를 철저히 지킬 것.
5. 코드 무결성: 출력 중간에 설명이나 인사말을 넣지 말고, 오직 마크다운
   코드 블록(```xml ... ```) 안에 완성된 SVG 코드만 출력할 것.
   닫는 태그(`</svg>`)를 잊지 말 것.
```

## 4. 생성 후 디버깅 — 수학적 수정 요청법

AI는 **눈으로 보는 것이 아니기 때문에** "눈이 찌그러졌어, 예쁘게 해줘"라고 하면 못 알아듣습니다.
**수학적/구조적인 언어**로 수정을 요청해야 합니다.

### 좌표 수정

| ❌ 나쁜 요청 | ✅ 좋은 요청 |
|-------------|------------|
| "오른쪽 눈을 좀 자연스럽게 옆으로 옮겨줘." | "`<!-- 🐱 오른쪽 눈 -->` 그룹의 `cx` 좌표를 현재 값에서 **+20**만큼 더해서 오른쪽으로 이동시켜 줘." |

### 레이어(Z-index) 수정

| ❌ 나쁜 요청 | ✅ 좋은 요청 |
|-------------|------------|
| "눈이 안 보여, 고쳐줘." | "눈이 머리통 뒤에 가려져 있어. **눈 그룹(`<g>`)의 코드**를 잘라내서 **머리통 코드보다 아래쪽 라인으로 옮겨서** 다시 출력해 줘." |

### 크기 수정

| ❌ 나쁜 요청 | ✅ 좋은 요청 |
|-------------|------------|
| "좀 더 크게 해줘." | "`<!-- 🐱 얼굴 -->` 그룹 내 원(circle)의 `r` 값을 현재 **60에서 80으로** 변경해줘." |

> [!TIP]
> **디버깅의 핵심 공식**: `어떤 그룹/요소` + `어떤 속성(cx, cy, r, width, height)` + `현재 값에서 ±N`
>
> AI를 **코더**로서 대우하고 **기하학적 제약**을 걸어주면,
> 저비용 모델로도 상업용 프로젝트에 쓸 수 있는 훌륭한 SVG를 무제한으로 생산할 수 있습니다.

## 5. WebCG-K에서의 활용

### AI SVG 생성 기능과의 연계

WebCG-K의 `callGeminiSvg()` 함수(Community C8)는 Gemini API를 통해 SVG를 자동 생성합니다.
위 가이드라인을 시스템 프롬프트에 통합하면 생성 품질이 대폭 향상됩니다.

```
webcg-k/src/lib/services/
  └── aiService.ts → callGeminiSvg()
       ├── 시스템 프롬프트: 위 가이드라인의 [엄격한 SVG 작성 규칙] 반영
       ├── 사용자 프롬프트: 요구사항 + 카테고리 (Logo, Background 등)
       └── 후처리: SVG 파싱 + Supabase Storage 업로드
```

### 가이드라인 적용 체크리스트

- [ ] 시스템 프롬프트에 `<path>` 최소화 규칙 추가
- [ ] viewBox를 `0 0 500 500`으로 고정하는 기본값 설정
- [ ] `<g>` 그룹화 + 주석 요구 규칙 추가
- [ ] 렌더링 순서(배경→전경) 명시 규칙 추가
- [ ] 색상 팔레트 제한 (사용자 선택 or 프로젝트 테마 색상)
