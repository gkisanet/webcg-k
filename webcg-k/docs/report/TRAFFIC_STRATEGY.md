### 3.4. Layer 3 — 실시간 이벤트 (비용: $25~100/월)

**핵심 전략: WebSocket 연결 수를 최소화하고, Broadcast 모델로 메시지를 팬아웃한다.**

#### 문제: 15만 동시 연결의 현실

Supabase Cloud Pro 플랜 기준:
```
기본 포함: 500 동시 연결
추가 비용: $10 / 1,000 연결

15만 동시 연결 = (150,000 - 500) × $10/1,000 = $1,495/월 (약 200만 원)
```

**→ 15만 동시 연결을 Supabase에 직접 붙이면 비용 폭발. 반드시 중간 계층이 필요.**

#### 해결: Fanout 아키텍처 (연결 수 1,000분의 1로 감소)

```
┌──────────────────────────────────────────────────────────────┐
│  KBS WebCG-K 컨트롤러                                        │
│  → Supabase Broadcast: "CG 송출" 이벤트 발행                  │
└────────────────────┬─────────────────────────────────────────┘
                     │ (연결 1개)
                     ▼
┌──────────────────────────────────────────────────────────────┐
│  Supabase Realtime (Broadcast 채널)                           │
│  구독자: Edge Relay 노드들 (5~10개)       ← 연결 5~10개      │
└────────────────────┬─────────────────────────────────────────┘
                     │
          ┌──────────┼──────────┐
          ▼          ▼          ▼
    ┌──────────┐ ┌──────────┐ ┌──────────┐
    │ Edge     │ │ Edge     │ │ Edge     │
    │ Relay 1  │ │ Relay 2  │ │ Relay 3  │
    │ (서울)   │ │ (부산)   │ │ (대전)   │
    │          │ │          │ │          │
    │ 5만 TV   │ │ 5만 TV   │ │ 5만 TV   │  ← 총 15만 TV
    └──────────┘ └──────────┘ └──────────┘
    Cloudflare    Cloudflare    Cloudflare
    Durable Object로 구현
```

> [!IMPORTANT]
> **Fanout 구조의 핵심**: Supabase에는 Edge Relay 노드만 연결 (5~10개). 각 Edge Relay가 5만 TV 연결을 담당. **Supabase 관점에서 동시 연결은 10개 미만** → 비용 $25/월 고정.

### 3.5. Layer 4 — 동적 데이터 API (비용: $0~5/월)

**핵심 전략: 보조 CG 상세 데이터는 Edge에서 캐싱한다.**

```typescript
// Cloudflare Worker — 보조 CG 데이터 API
export default {
  async fetch(request, env) {
    const url = new URL(request.url);
    const cgId = url.searchParams.get("id");
    
    // 1. Edge Cache에서 먼저 조회 (캐시 히트 시 오리진 접근 없음)
    const cacheKey = `cg:${cgId}`;
    const cached = await env.KV.get(cacheKey, "json");
    if (cached) return Response.json(cached);
    
    // 2. 캐시 미스 시 Supabase에서 조회 (드물게 발생)
    const { data } = await supabase
      .from("supplementary_cg")
      .select("*")
      .eq("id", cgId)
      .single();
    
    // 3. Edge KV에 캐시 저장 (TTL: 5분)
    await env.KV.put(cacheKey, JSON.stringify(data), { expirationTtl: 300 });
    
    return Response.json(data);
  }
};
```

## 6. 클라이언트 측 최적화

### 6.1. TV 앱 내 로컬 캐싱

```typescript
// TV 앱의 보조 CG 로컬 캐시 전략
class CgCache {
  private cache: Map<string, { data: any; expiry: number }> = new Map();
  
  // 메모리 캐시 (세션 중)
  get(id: string): any | null {
    const entry = this.cache.get(id);
    if (entry && Date.now() < entry.expiry) return entry.data;
    this.cache.delete(id);
    return null;
  }
  
  set(id: string, data: any, ttlMs: number = 300_000): void {
    // 캐시 크기 제한 (TV 메모리 보호)
    if (this.cache.size >= 50) {
      const oldest = this.cache.keys().next().value;
      this.cache.delete(oldest);
    }
    this.cache.set(id, { data, expiry: Date.now() + ttlMs });
  }
}
```

### 6.2. 적응형 폴링 (Adaptive Polling)

```typescript
// 방송 시간대에만 활발하게 폴링, 비방송 시간에는 절전
const POLLING_INTERVALS = {
  ACTIVE: 2_000,    // CG 활성 상태: 2초
  IDLE: 10_000,     // 방송 중이지만 CG 없음: 10초
  STANDBY: 60_000,  // 비방송 시간: 1분
} as const;

function getPollingInterval(state: AppState): number {
  if (state.hasActiveCg) return POLLING_INTERVALS.ACTIVE;
  if (state.isLiveBroadcast) return POLLING_INTERVALS.IDLE;
  return POLLING_INTERVALS.STANDBY;
}
```

### 6.3. 네트워크 트래픽 최소화

| 기법 | 효과 | 구현 |
|------|------|------|
| **조건부 요청 (ETag)** | 변경 없으면 304 (본문 0바이트) | `If-None-Match` 헤더 |
| **gzip/brotli 압축** | JSON 60-80% 크기 감소 | Cloudflare 자동 적용 |
| **증분 업데이트 (diff)** | 전체 데이터 대신 변경분만 전송 | JSON Patch |
| **프리페치** | 다음 보조 CG 사전 로드 | 큐시트 순서 예측 |
