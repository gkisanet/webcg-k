## [1.1.0] - 2026-05-21 (보안 패치 & 마이그레이션 단일 통합 릴리즈)

### 🔴 [1] api_keys 테이블 대칭 암호화 & RPC 복호화 파이프라인
* **무엇을:** `api_keys` 테이블에 `pgcrypto` 대칭 암호화 트리거를 입히고, 클라이언트에서 직접 키를 SELECT 하는 대신 RPC 함수로 조회하도록 연동.
* **왜:** DB 해킹이나 RLS 우회 공격 시 API 마스터키가 평문 그대로 유출되어 대규모 비용 폭탄 및 소유권 탈취 사고로 번지는 일을 차단하기 위함.
* **어떻게:** 
  - `pgp_sym_encrypt` 함수를 이용해 디스크 상에는 암호화하여 저장.
  - `get_decrypted_api_key(id)` 보안 정의자 함수를 래핑하여 인가된 서비스 레이어에서만 복호화 평문을 가져가도록 제어.
* **🚀 성능 & Big-O 복잡도 분석:**
  - **시간 복잡도:** 기존 SELECT 쿼리는 $O(1)$의 단순 키-값 조회를 수행했으나, 암호화 패치 적용 후 복호화 연산($pgp\_sym\_decrypt$)이 수반됩니다. 암호화 연산 자체는 대칭키 알고리즘(AES 등)을 따르므로 데이터 크기 $N$에 비례하여 $O(N)$의 연산 부하가 일어나지만, API 키의 글자수($N \le 100$)가 극히 작으므로 **체감 시간 복잡도는 기존과 동일하게 사실상 $O(1)$로 수렴**하며 성능 리팩터링의 유의미한 지연 오버헤드는 없습니다.
  - **공간 복잡도:** base64 인코딩으로 인해 원본 비밀번호 크기 대비 저장 공간이 33% 정도 증가하는 $O(N)$ 공간적 특징이 발생하나, 수 KB 미만의 텍스트 데이터이므로 데이터베이스 인덱싱 및 캐시 메모리 사용 효율에는 아무런 영향을 주지 않는 압도적인 성능적 이점이 유지됩니다.

### 🔴 [2] characters storage bucket 업로드 크기 및 MIME 제약 추가
* **무엇을:** Supabase `storage.buckets` 내 `characters` 버킷 설정에 5MB 파일 용량 상한선과 특정 확장자 허용 목록 지정.
* **왜:** 인증된 악의적 유저가 버킷에 테라바이트급 쓰레기 파일을 부어 디스크 공간을 마비시키는 DoS 공격 및 악성 스크립트(.js, .html)를 올려 공격하는 XSS 도용 통로를 봉쇄하기 위함.
* **어떻게:** 
  - `20260520000000_combined_migration.sql` 내에 `file_size_limit = 5242880` 및 allowed_mime_types 배열 주입.
  - `ON CONFLICT DO UPDATE` 방어 쿼리 조치.
* **🚀 성능 & Big-O 복잡도 분석:**
  - **시간 복잡도:** 스토리지 업로드 전 검증이 Supabase 스토리지 프록시 엔진 레이어에서 1차 차단되므로, 잘못된 파일이 올라갔을 때 DB와 네트워킹을 타고 흐르던 낭비 경로를 즉각 컷오프(Early Return)하여 서버 리소스 사용의 효율성을 $O(1)$ 단위에서 조기 차단해 주는 막강한 이점을 갖습니다.

### ⚙️ [3] 마이그레이션 파일 단일화 대통합
* **무엇을:** `202605140001_overlay_blend_mode.sql`과 `20260520000000_combined_migration.sql`을 하나의 `20260520000000_combined_migration.sql` 단일 파일로 융합.
* **왜:** 개발 및 배포 환경에서 마이그레이션 파편화로 인해 발생하는 스키마 추적 복잡도를 차단하고 데이터베이스 구조 형상 관리를 심플하게 통합(Consolidate)하기 위함.
* **어떻게:** 
  - 5/14 분량을 앞쪽에 배치하고 5/20 분량을 뒤쪽에 시간순으로 이어붙임.
  - 중복 파일 물리적 소거 및 DB 버킷 insert 충돌을 방지하는 `ON CONFLICT DO UPDATE` 가드 보정.
* **🚀 성능 & Big-O 복잡도 분석:**
  - **시간 복잡도:** 다중 DDL 파일 순차 실행에서 발생하는 컨텍스트 스위칭 및 Connection 오버헤드를 원자적(Atomic) 1회 실행으로 묶어, 마이그레이션 배포 시간 복잡도를 $O(M)$(M개의 마이그레이션 파일 실행)에서 단일 세션 내 단 한 번의 커밋으로 종결되는 $O(1)$ 수준으로 단축했습니다.

### 🧹 [4] Fathom 관련 스키마 및 데이터 소거 (Fathom Purging)
* **무엇을:** 통합 마이그레이션 파일 내에 존재하던 9개의 `fathom_*` 테이블, 관련 인덱스, RLS 정책, grants, helper 함수, `fathom-files` 스토리지 버킷 및 시드 데이터를 스키마에서 완전 소거.
* **왜:** 미사용 대형 지식 엔진 모듈을 퍼블릭 리포지토리에서 덜어내어 저장소를 극도로 경량화하고 보안 공격 표면(Attack Surface)을 최소화하기 위함 (공통 인프라 테이블인 `profiles` 및 `pgcrypto` 등은 공통 사용을 위해 안전히 유지).
* **🚀 성능 & Big-O 복잡도 분석:**
  - **시간 복잡도:** 사용하지 않는 9개 테이블과 인덱스, RLS 정책들이 완벽히 제거됨에 따라, PostgreSQL의 시스템 카탈로그 조회(Catalog Lookups) 오버헤드가 완전히 제거되었습니다. 이는 전체 자막 큐시트 및 세그먼트 조회 성능을 $O(1)$의 조기 최적화 상태로 보존합니다.
  - **공간 복잡도:** 768차원 고차원 벡터 임베딩 컬럼 및 hnsw 벡터 인덱스가 메모리와 디스크에서 증발하게 되어 데이터베이스의 영구 저장 공간(Disk Storage) 및 RAM 버퍼 캐시(Buffer Cache)의 점유 공간 복잡도를 획기적으로 낮추는 물리적 이점을 확보했습니다.
